## TF-NON-XLA-SHAPE-001 - Non-XLA Translation Batch Dimension Mismatch
**Category:** TensorFlow, Translation, Shape Handling
**Impact:** TensorFlow training/inference with gridsize > 1 when XLA disabled (`USE_XLA_TRANSLATE=0`)
**Location:** ptycho/tf_helper.py:702-839 (`translate_core`, `Translation.call`)
**Date:** 2025-11-14

### Issue
When XLA is disabled (`USE_XLA_TRANSLATE=0`) and gridsize > 1, TensorFlow training crashes during the first epoch with shape mismatch in `translate_core`:
```
Shapes of all inputs must match: values[0].shape = [4] != values[2].shape = [128]
 [[{{node functional_1/padded_objs_with_offsets_1/translation_36_1/stack_1}}]]
```
**Root cause:** The non-XLA `ImageProjectiveTransformV3` fast path builds transformation matrices using `tf.stack` on tensors (`ones`, `zeros`, `dx`, `dy`) that must all have the same batch dimension. When `gridsize > 1`, `_channel_to_flat` expands images from shape `(b, N, N, c)` to `(b*c, N, N, 1)`, but `translations` remains `(b, 2)`. The mismatch causes `tf.stack` to fail.

### Fix (2025-11-14)
**Phase 1 (Immediate):** Disabled the `ImageProjectiveTransformV3` fast path entirely when `use_xla=False`. The function now always uses the fallback `_translate_images_simple` implementation for non-XLA mode, which includes broadcast logic to tile translations to match the images batch dimension via `tf.repeat`.

**Location:** ptycho/tf_helper.py:735-753, 756-839
- Lines 742-744: Compute `images_batch` and `trans_batch` for guard check
- Lines 746-753: Skip fast path when `use_xla=False` (unconditional fallback)
- Lines 813-828: Broadcast translations using `tf.cond` and `tf.repeat` when batch dimensions mismatch

**Rationale:** Attempting to conditionally use the fast path in graph mode (via `tf.cond` or `tf.debugging.assert_equal`) introduced complexity:
- `tf.debugging.assert_equal` raises at graph execution time, bypassing `try/except`
- Nested `tf.cond` with the existing XLA path adds control flow complexity
- The performance benefit of `ImageProjectiveTransformV3` vs `_translate_images_simple` is minor for typical batch sizes
- Disabling the fast path entirely when `use_xla=False` is the simplest, most reliable solution

## REPORTING-ARTIFACT-BOUNDARY-001 - Optional Reporting Artifacts Must Not Decide Scored Experiment Failure
**Category:** Experiment Orchestration, Artifact Contracts, Robustness
**Impact:** Long-running study loops can misclassify successful experiments as crashes and waste debug budget
**Date:** 2026-03-29

### Rule
For scored experiment workflows, only core execution evidence should determine
`KEEP`/`DISCARD`/`TIMEOUT`/`CRASH`:
- launcher exit status
- primary metrics
- any required comparability contract such as randomness metadata

Optional reporting artifacts such as comparison galleries, probe-inclusive
rerenders, or convenience visual exports may add warnings, but they must not by
themselves change a successful scored run into `CRASH`.

### Why
Post-processing and publication helpers are much more brittle than the actual
training/inference path. If they are treated as fatal, the workflow can:
- stop autonomous search even though the experiment finished
- trigger pointless crash-debug attempts against candidate code
- hide the real experiment outcome by failing after metrics were already
  produced

### Recommended Pattern
- Persist launcher result before later harvest or publication logic.
- Classify from core artifacts first.
- Record optional publication status separately, for example
  `published`, `fallback_plain_compare`, or `missing_nonfatal`.
- Surface optional-artifact problems as warnings in run/assessment artifacts.

### Status
**Recommended practice** — adopted for the `lines_256` controller path

## RECTANGULAR-SCALED-001 - Rectangular-Scaled Physics Forward Mode Replaces the Amplitude Scaling Stack
**Category:** PyTorch, Physics Forward Model, Main-Branch Reconciliation
**Impact:** `ModelConfig.physics_forward_mode='rectangular_scaled'` (opt-in; default `'amplitude'` path is unaffected)
**Location:** `ptycho_torch/model.py:1356-1435` (`RectangularScaledDiffraction`), `ptycho_torch/model.py:1698-1731` (`RectangularPoissonLoss`/`RectangularMAELoss`), `ptycho_torch/model.py:2156-2270` (loss/scale-routing dispatch), `ptycho_torch/config_params.py:96-161`
**Date:** 2026-07-02

### What Changed
Backlog item B5 (`docs/plans/2026-06-29-main-fno-stable-physics-reconciliation-backlog.md`)
ports `main`'s rectangular forward physics onto `fno-stable` as an opt-in mode. When
`physics_forward_mode='rectangular_scaled'`, the entire default-path scaling stack —
`ProbeIllumination` -> `pad_and_diffract` -> `IntensityScalerModule.inv_scale` -> a
loss-time `physics_scale` multiply — is **replaced**, not layered on top of, by
`RectangularScaledDiffraction` (ported verbatim from `main`) plus `main`'s
`modified_output_scale` routing: `compute_loss` folds `probe_scaling` (`batch[2]`) and
`physics_scale` directly into the forward's `output_scale_factor` as
`sqrt(1/(scale^2 * physics_scale + 1e-9))` (`model.py:2259-2269`), instead of applying a
separate multiply after the forward pass. `RectangularScaledDiffraction` requires
`real_imag`-derived object patches (`cnn_output_mode='real_imag'`, Task 2.3 / backlog B1)
because FT-linearity of the real/imag representation is what makes the predicted
intensity quadratic in the two new scale scalars `s1`/`s2`; the constructor raises
`ValueError` for amp/phase-derived objects (`model.py:1862-1870`).

### Physical Semantics of s1/s2 (Corrected, 2026-07-02)
`s1`/`s2` are the **dynamic amplitude/photon-scale factorization** described in the
PtychoPINN-CI manuscript "Contrast-invariant deep ptychography neural networks" (Vong,
Henke, Hoidn, Mehta, Shapiro, Hexemer, Schwarz), Eq. 1/3/5 — **not** a decoder-box
workaround (see Retraction below). The `real_imag` decoder emits unit-less tanh textures
`a~`, `b~` in `[-1, 1]`; the physical object is `O = s1*a~ + j*s2*b~`. Because rectangular
coordinates are linear through the FFT, the predicted intensity is quadratic in the two
scalars:

```
I = |s1*Psi_a + s2*Psi_b|^2 = s1^2|Psi_a|^2 + 2*s1*s2*Re[conj(Psi_a)*Psi_b] + s2^2|Psi_b|^2
```

(code: `RectangularScaledDiffraction.forward`, `model.py:1379-1424`). Physical reading:
`c_A = sqrt(s1^2 + s2^2)` is the amplitude/photon-count scale, `c_phi = arctan(s2/s1)` is
an intrinsic phase-contrast angle. Primary role: `s1`/`s2` absorb the *absolute*
photon/amplitude scale of a given dataset, decoupling the learned real/imag texture from
per-dataset measurement scale — the manuscript's headline result is training one network
that reconstructs correctly across datasets spanning orders of magnitude in photon flux
without retraining.

**Verified facts (this session, code-level):**
- `s1`, `s2` scale by `sqrt(K)` when measured intensity is scaled by `K` (tested `K=4` ->
  `2.000x`, `K=100` -> `10.000x`, exact) — consistent with `I ~ s^2` in the quadratic form
  above.
- The scale is pinned via the Poisson NLL at training (`PoissonLoss`/`RectangularPoissonLoss`
  fit `pred` to absolute counts) and via the inference least-squares solve
  (`solve_scaling_factors_from_basis` + `enforce_physics_constraint`, `model.py:1462-1515`,
  Eq. 6 of the manuscript). An MAE/L1 loss is (near) scale-free and would not pin `s1`/`s2`.
- Inference asymmetry: `forward_predict` (`model.py:2197`) applies only the RMS
  `input_scale_factor`, not the output/physics scale; `compute_loss` applies
  `modified_output_scale`. So at inference, `s1`/`s2` (re-solved against raw counts) carry
  the *entire* output scale; at training they sit atop `modified_output_scale`.
- Units are dataset-dependent: the solve/loss operate on whatever intensity values the NPZ
  stores (raw photon counts for real/synthetic count-convention datasets).

**Validated flux-sweep result (photon-flux sweep, one frozen CNN gs1 checkpoint trained at
mean-count 100, evaluated at mean-count {1, 100, 10000}):**
- `c_A` = 0.913 / 9.279 / 92.96 -> ratios vs. mean=100 = 0.098 / 1.000 / 10.019 vs.
  expected `sqrt(flux)` = 0.100 / 1.000 / 10.000 (**exact** `sqrt(flux)` scaling).
- `c_phi = arctan(s2/s1)` = -64.9 / -65.8 / -65.4 deg (flux-**invariant** — the intrinsic
  phase-contrast reading holds regardless of measurement flux).
- Reconstructed `|O|`: varpro-OFF stays flux-invariant (~1.0, the RMS-normalized decoder
  texture); varpro-ON scales as `sqrt(flux)` (0.72 / 7.6 / 76).

**Count-unit caveat:** varpro-ON `|O|` lands in **count-amplitude units**, not the
manuscript's `~1`, because this pipeline normalizes the probe at inference time, whereas
the manuscript's Sec. 2.2 leaves the probe un-normalized so the probe itself carries flux
and `s1`/`s2` stay near `1`. Any amplitude-vs-truth comparison across this pipeline's
varpro-ON canvases must divide out `c_A` before comparing to a normalized-truth object.

**Retraction:** an intra-session hypothesis that "trainable `s1`/`s2` help when strong
phase binds the `real_imag` decoder box" is **wrong and retracted**. `s1`/`s2` are two
global-per-dataset scalars; they cannot rescue individual box-railed pixels, and box
representability (below) is orthogonal to scaling. `rect_s1s2_trainable` (the train-time
knob) is secondary — inference always re-solves `s1`/`s2` via the least-squares path
regardless of that flag; the primary value proposition is inference-time varpro ON vs.
OFF cross-scale generalization (manuscript Fig. 3c), not train-time frozen-vs-trainable.

### Representation Restriction: Why real_imag Only
`s1`/`s2` scale the real and imaginary channels independently — a representation-space
scaling with two separable degrees of freedom. The polar (`amp_phase`) analogue of that
scaling is a single amplitude scale; there is no two-DOF decomposition of an
amplitude/phase-parameterized output that corresponds to independently scaling `s1`/`s2`.
This is why `physics_forward_mode='rectangular_scaled'` requires `cnn_output_mode='real_imag'`
rather than being representation-agnostic.

### Loss-Layer Parity Quirk: MAE Re-Square (Replicated, Not a Bug)
`main`'s `MAELoss` squares its `pred` argument before the L1 comparison
(`mae(pred**2, raw)`) on the assumption that `pred` is an amplitude. Because
`RectangularScaledDiffraction` already returns an intensity, `main`'s MAE path
double-squares the prediction. `RectangularMAELoss` (`model.py:1716-1731`) reproduces this
**verbatim** for frozen-fixture parity — this is a **documented, deliberate** parity
choice (amendment #2/#13 in the reconciliation ledger), not a defect to fix. Contrast
`RectangularPoissonLoss` (`model.py:1698-1713`), which replicates `main`'s
`PoissonIntensityLayer` semantics exactly: the Poisson rate is `pred` directly, and
neither `pred` nor `raw` is re-squared. The default `'amplitude'` path's
`PoissonLoss`/`MAELoss` are byte-for-byte unchanged either way.

### Decoder Box Representability Constraint (Corrected Bounds)
The `real_imag` CNN heads use `main`'s hardwired `ScaledTanh` activations
(`model.py:590-599`, ported verbatim): real via `tanh + 0.2` giving range `(-0.8, 1.2)`,
imag via `1.2 * tanh` giving range `(-1.2, 1.2)`. **Correction:** an earlier figure of
`0.65` for the imag ceiling (propagated through an intra-session flux-sweep-era analysis)
was wrong; the reviewer-verified correct bound is `+/-1.2`. This does not affect any
validated result above — the `s1`/`s2` photon-scale and flux-sweep findings never
depended on the box. The representability claim still holds via the **real floor**
(`-0.8`): a unit-amplitude object at `|phase| -> pi` maps to `real ~ -1`, below the floor,
so it is unreconstructable in `real_imag` mode. `amp_phase` mode (`pi*tanh` phase head)
has no such ceiling and is the correct choice for high-phase-contrast objects. This
decoder-box constraint (backlog B1/B2, Task 2.3) is **separate** from the `s1`/`s2` scale
factorization above — see Retraction.

### C=1-vs-Main Residual Differences (object_big Padding)
Cross-branch parity verification (Task 2.8, `test_cross_branch_rectangular_parity.py`,
commit `95bcde03` + fix `82da7796`) established:
- `C=1, object_big=False`: forward and loss values match the frozen `main`-parent
  fixtures within the registered cross-build contract (`rtol=1e-5, atol=1e-6`)
  under fno-stable's **real, current** padding. The object-big=False path does not
  touch the reassembly canvas, so this is the rectangular forward parity gate.
- `C=1, object_big=True`: under fno-stable's actual `buffer=0`
  (`ptycho_torch/helper.py::get_padded_size`,
  commit `ba3f705d`, shared by both the amplitude and rectangular forward paths,
  predates this reconciliation), the frozen fixture has a finite padding residual.
  A `get_padded_size` monkeypatch that restores `main`'s padding buffer brings the
  forward and loss results within the same cross-build tolerance. Reverting that
  buffer would break the Task 2.1 regression pin, so it is not reverted here. The
  `object_big=True` (bigT) rectangular path remains tracked as an open backlog item
  ("bigT REAL-padding parity") in
  `docs/plans/2026-06-29-main-fno-stable-physics-reconciliation-backlog.md`.

### Status
Active. `physics_forward_mode='rectangular_scaled'` is math-verified (fixture parity) and
end-to-end trainable (Task 2.8 independent smoke: finite loss, nonzero data-dependent
gradients on all params including `s1`/`s2`). Remaining open items (padding parity,

## REASSEMBLY-BRIDGE-001 - Four Traps Evaluating Grid-Lines Checkpoints via reconstruct_image_barycentric

> **Scope superseded 2026-07-09:** The observations below remain valid evidence for `legacy_v1`/`normalized_amplitude` checkpoints. They do not define CI absolute units. `ci_intensity_v2` uses `probe_physical`, the training detector mask, full-frame basis accumulation, and no training output scale; see CI-ABSOLUTE-SCALE-CONTRACT-001.
**Category:** PyTorch, Reassembly, Cross-Pipeline Bridging, VarPro Calibration
**Impact:** Evaluating grid-lines-workflow-trained checkpoints through `ptycho_torch.reassembly.reconstruct_image_barycentric` (inference-side bridging, ablation harnesses); any future NPZ bridge from `grid_lines_workflow`'s schema into `PtychoDataset`'s `xcoords`/`ycoords` contract
**Location:** `ptycho/workflows/grid_lines_workflow.py:538-579` (`_build_scan_positions`), `ptycho_torch/dataloader.py:640-642`, `ptycho_torch/reassembly.py:874-878` (`VectorizedWeightedAccumulator.valid_mask`), `ptycho_torch/reassembly.py` (`compute_varpro_basis`), `ptycho_torch/config_params.py:253` (`InferenceConfig.window`)
**Date:** 2026-07-02

### Issue
A multi-agent investigation into cross-pipeline reassembly (bridging grid-lines-workflow-trained checkpoints' test predictions into `reconstruct_image_barycentric` for evaluation outside the grid-lines runner's own stitching path) surfaced four distinct traps/bugs on the way to a working bridge. None were previously documented in one place; recording them here so a future bridging task does not rediscover them independently.

### Root Cause
1. **Coordinate transpose trap.** `_build_scan_positions` (`grid_lines_workflow.py:538-579`) stores `coords_global[:,0,0,:]=iy` (row) and `coords_global[:,0,1,:]=ix` (col), but torch loaders stack `xcoords`/`ycoords` in x,y order into `coords_global` (`dataloader.py:640-642`: `np.stack([xcoords, ycoords], axis=1)`), which `reconstruct_image_barycentric` then reads as `xmin = positions_px[:,0]-half_size`, `ymin = positions_px[:,1]-half_size`. A bridge that naively assigns `xcoords=coords_offsets[...,0]` transposes every scan position: amp_corr is stuck at 0.075 (`0.0752` measured) until the assignment is swapped, jumping to 0.52 (`0.5239`) once corrected — a pure axis-relabeling fix, no geometry change on either side. Fix: assign `xcoords = coords_offsets[..., 1]`, `ycoords = coords_offsets[..., 0]` when bridging this schema.
2. **Border dead ring (scoring artifact, not a placement bug).** `VectorizedWeightedAccumulator.valid_mask` (`reassembly.py:874-878`) requires `xmin_wh + patch_size + 1 < canvas_shape[1]` (a strict `+1` margin) and drops any patch failing it entirely (41 of 441 scan-position rows here, `total_valid_patches=400`) — canvas size is built to snugly fit exactly the outermost scan positions with zero slack, so this discards the outermost ring of positions outright, leaving a genuine ~20px near-zero-weight edge. Scoring the whole canvas against truth (which has real, nonzero content in that ring) silently caps correlation at ~0.5-0.6 regardless of prediction quality: an oracle (ground-truth-patch, no-model) reconstruction plateaus at the *same* 0.6036 whole-canvas correlation, and the oracle's and the model's zero-weight pixel masks are pixel-for-pixel identical (`IoU=1.0`) — proving the ring is a pure function of scan-position geometry, independent of patch content. Once the ring is cropped (20px), oracle interior correlation is `0.99999999987` (numerically exact placement) and the real model reaches 0.9744-0.9856 (see table below). Workaround: crop the ring before scoring; `InferenceConfig.window` (`config_params.py:253`, default `20`, docstring "Window padding around reconstruction due to edge errors") is the designed hook — the evaluation harness (`varpro_probe_ablation_runner.compute_metrics`) currently never applies it.

## TORCH-REASSEMBLY-SIGN-001 - Torch object_big Reassembly Offset-Sign Convention Corrected to Match TF (Behavior Change + Reversal Recipe)
**Category:** PyTorch, Coordinate Convention, Behavior Change

### Issue
`ptycho_torch/helper.py::reassemble_patches_position_real` (and `_probe`) placed grouped patches on the shared bigN canvas by translating content by `+offset`, while the TF reference `ptycho/tf_helper.py::_reassemble_patches_position_real` (tf_helper.py:909, `Translation([imgs, -offsets_flat])`) — the implementation whose conventions generated every dataset's coordinates — translates by `-offset`. `extract_channels_from_region` was the exact inverse (`-offset`), so merge→extract round-trips perfectly and the inconsistency is invisible to any internal-consistency test. Discovered 2026-07-06 when cnn@gs2 failed its pipeline-validation gate: native amp MAE 0.7377 (vs 0.1402 cnn@gs1 reference), reconstruction a fine periodic checkerboard at exactly the ±2px group-offset scale.

### Root Cause / Evidence
The two conventions differ by a point reflection of each gridsize>1 group through its center: the patch belonging up-right of the group center is placed down-left, etc. The measured diffraction pins each patch's content to the true object at its position, but the mirrored merge forces overlap-averaging between the wrong patch edges — the data and the overlap constraint become contradictory, and the optimizer settles into the observed periodic compromise. At gridsize 1 the per-group relative offset is identically `position − mean(position) = 0`, so both conventions coincide bit-for-bit — every gs1 baseline, test, and checkpoint is unaffected under either sign.

Decisive probe (single bright pixel, offset `(dx=+5, 0)`, C=1 group, symmetric ±offsets to neutralize TF batch-mean recentering): torch moved content +5 columns, TF moved it −5 columns on identical input — for BOTH torch branches (object_big merge AND the `crop_size` inference-stitch branch) against BOTH TF reassemblers (`_reassemble_patches_position_real` and `shift_and_sum`/`reassemble_position`), which agree with each other. Full chain (incl. ruled-out x/y-transpose hypothesis and a superseded mis-shaped-offsets probe): `.superpowers/sdd/ext/task-gs2-t3-report.md`, `.superpowers/sdd/ext/task-gs2-t3b-report.md`.

## RECT-MAE-UNITS-001 - rectangular_scaled Unit Contract Under the Grid-Lines Bridge (MAE Double-Square Collapse; No Count Calibration)

> **Scope superseded 2026-07-09:** This is the explicit legacy rectangular MAE contract and its historical failure evidence. New rectangular workflows default to CI count intensity, reject MAE before loading, and no longer depend on bridge `physics_scaling_constant`; see CI-ABSOLUTE-SCALE-CONTRACT-001.
**Category:** PyTorch, Physics Forward Model, Unit Contract, Bridge Scale Constants
**Impact:** `physics_forward_mode='rectangular_scaled'` training through the grid-lines runner bridge under either loss mode; interpretation of the s1/s2 scale factors and the historical s1≈131 signature; scope of any "poisson-native" rect test
**Location:** `scripts/studies/grid_lines_torch_runner.py:1293-1299`, `ptycho_torch/workflows/components.py:93,262,322,519-529`, `ptycho_torch/model.py:1691,1698-1713,1716-1731,2259-2267,2283`
**Date:** 2026-07-06

### Issue
Two questions blocked a fair poisson-native test of the `rectangular_scaled` forward on the aligned grid-lines data: (a) why does `rectangular_scaled`+MAE collapse to a dot lattice (HYBRES-ALIGN-001, RECTANGULAR-SCALED-001), and (b) what `diffraction` unit convention would a `rectangular_scaled`+poisson run require. Task 5a (`.superpowers/sdd/ext/task-ladder-t5a-report.md`, a CPU read-only trace of the real `_build_lightning_dataloaders` on the runner's dict container) answered both and descoped the count-level rect-poisson rung.

### The MAE Double-Square Collapse
`RectangularScaledDiffraction` returns an **intensity** `I_pred = Σ|FFT(exit_wave)|²`. `RectangularMAELoss` (`model.py:1716-1731`) replicates main's `MAELoss` quirk verbatim — it squares `pred` before the L1 compare (`mae(pred**2, raw)`). Because `pred` is already an intensity this **double-squares** the prediction, while `raw` is the NPZ `diffraction`, a normalized **amplitude** (DATA-001 / NORMALIZATION-001). The effective objective is therefore `abs(I_pred² − A_meas)` — a fourth-power prediction fit against a first-power amplitude observation. Its fixed point (`I_pred² = A_meas`, i.e. `I_pred = A_meas^{1/2}`) drives the predicted intensity toward a near-flat spectrum, which inverse-transforms to the dot-lattice / near-constant canvas collapse seen in the ablation. Trained `s1`/`s2` stayed ≈1 throughout (checkpoint evidence, RECTANGULAR-SCALED-001) because this near-scale-free objective does not pin them. Contrast fno-stable's default amplitude-domain `MAELoss` (`model.py:1691`), which compares amplitude-to-amplitude, and `RectangularPoissonLoss` (`model.py:1698-1713`), which squares **neither** side. The quirk is a deliberate frozen-fixture parity choice, not a fixable bug (RECTANGULAR-SCALED-001).

### Corrected Unit Contract (supersedes REVISION R1's amplitude-path mechanism)
REVISION R1 asserted the amplitude path "count-lifts internally" (physics_scale≈328.7 multiplying pred and obs, `model.py:2290-2291`). This is **false for the grid-lines runner bridge.** The runner builds a plain dict container (`grid_lines_torch_runner.py:1293-1299`) with no `physics_scaling_constant`/`scaling_constant`/`rms_scaling_constant`; `_attach_physics_scale` (`components.py:93`) is reachable only via `_ensure_container` (`components.py:262,322`), which the dict bypasses, so `_select_scale(None)` returns 1.0 (`components.py:519-529`). **physics_scale = rms_scale = scale = 1.0 in every ladder rung, both loss modes** (toy-verified against the real dataloader). Consequences for the rect path:
- `output_scale = sqrt(1/(scale²·physics_scale + 1e-9)) = 1.0` (`model.py:2259-2267`); the rect forward is **Parseval-locked to O(1) normalized intensity** with no absolute photon-count calibration, and the rect loss branch applies no post-hoc pred·physics_scale multiply (`model.py:2283`).
- `s1`/`s2` are the only absolute-scale degrees of freedom in the rect forward (`I_pred ∝ s1²|Ψ_a|² + s2²|Ψ_b|² + 2 s1 s2 Re[Ψ_aΨ_b*]`).
- A well-posed Poisson NLL needs rate and obs in the same intensity/count units. Of the three candidate conventions, only `data_intensity=(amp·S)²` is domain-correct, but with `output_scale=1` it forces `s1²≈counts/energy` ⇒ s1≈O(100) — the historical near-collapse regime, not a fair test. The row that keeps s1/s2 O(1) is a **normalized-intensity** twin `amp²` (Task 5a row d), which is itself a normalized-units NLL, not a count Poisson.
- **A genuine count-level `rectangular_scaled` poisson test is infeasible through the unmodified bridge with any dataset convention.** The minimal enabling change — populate the scale constants on the bridge container in `ptycho_torch/workflows/components.py` — alters scale for **all** bridge training (including the amplitude rungs) and requires its own authorized plan plus full revalidation. Rung 3 is descoped on these grounds (plan REVISION R2).

### s1≈131 and the ~53×-Low VarPro s1 Are the Same Defect
The historical `s1=131.65` (from `ext_matrix/hybres_gs1_both/probe_varpro`) is a **VarPro inference solve** on a near-collapsed canvas, not a trained weight. With the rect forward Parseval-locked to O(1) and no output-side count lift, the scale solve absorbs the entire gap to count-intensity observations; the Task 5a toy predicts s1≈104 for `(amp·S)²` with S≈328.7, and a near-zero canvas energy raises the solved value to ≈131. The REASSEMBLY-BRIDGE-001 #4 open item — VarPro s1≈0.0886, ~53× **below** the sensible `norm_Y_I≈4.7` scale — is the **same** unpinned rectangular-scale defect in the **opposite** direction: there `PtychoDataset` stages a different intensity convention, so the least-squares solve lands low instead of high. Both are one root cause: the rect path has no canonical absolute-scale calibration in these bridges, and s1/s2 (or the VarPro solve) silently absorb whatever units mismatch the staging presents.

### Status
Active. `rectangular_scaled`+MAE collapse is a construction artifact of the double-square objective (parity is deliberate — not a bug to fix). Count-level rect-poisson is descoped pending a bridge-scale-constant plan. Cross-refs: RECTANGULAR-SCALED-001 (s1/s2 semantics, MAE-quirk provenance), HYBRES-ALIGN-001 (rect+MAE collapse under the aligned regime), REASSEMBLY-BRIDGE-001 #4 (the ~53× VarPro units item this explains), POISSON-NORM-001 / POISSON-LADDER-001 (the amplitude-path rungs).

## POISSON-NORM-001 - Count-Scale Diffraction Collapses Amplitude-Path Poisson Training (NORMALIZATION-001 Negative Control)
**Category:** PyTorch, Normalization, Data Contract, Poisson Objective
**Impact:** Any amplitude-path poisson training fed pre-count-scaled `diffraction`; the load-bearing status of NORMALIZATION-001 and the dataloader "must be normalized" comment
**Location:** `ptycho_torch/dataloader.py:698-705`, `ptycho_torch/workflows/components.py:519-529`, `ptycho_torch/helper.py:723-749`, `.artifacts/varpro_ablation/poisson_ladder/{rung1_cnn,rung1_cnn_lr1e3}`
**Date:** 2026-07-06

### Issue
The Poisson Validation Ladder's first attempt fed a count-convention "twin" dataset (`data_amp.diffraction = amplitude·S`, S≈328.7, so the squared observation is genuine ~1e9-count intensity) to cnn amplitude-path poisson training, on the (later-refuted) premise that poisson needs count-scale observations. It collapsed. It is retained as a documented **negative control**: it proves NORMALIZATION-001 is load-bearing, not advisory.

### Mechanism (REVISION R2 — corrected)
In the grid-lines bridge the scale constants are never populated, so `physics_scale = 1.0` for **both** the original normalized data and the count twin (there is **no** 328.7→1.0 degeneracy — see RECT-MAE-UNITS-001 for the corrected bridge model). The twin collapsed because its count-scale observations (`(amp·S)²` ~O(10⁴)/px as intensity) are **unreachable by the bounded network forward at scale 1.0**: the `−k·logλ` term of the Poisson NLL dominates and drives the output to a constant. On the **native container path** (`PtychoDataContainerTorch`/`RawData`, which does call `_attach_physics_scale`) the same count data would **additionally** degenerate `derive_intensity_scale_from_amplitudes` (`helper.py:723-749`) to 1.0, because `mean(sum(X²)) = nphotons` by construction — a second, path-specific failure. Either way, count-scale `diffraction` breaks amplitude-path poisson training: NPZ `diffraction` must remain normalized amplitude, with `nphotons` carried as metadata only (NORMALIZATION-001, `dataloader.py:698-705`). The loader accepts count-scale input without an assertion (a defensive loader guard is a separate, protected-adjacent change — out of scope here, noted for the record).

### Evidence
Both recipes converged to a byte-identical constant-output attractor within epoch 1 and never escaped through 5 epochs:
- primary (lr 2e-4, ReduceLROnPlateau) and fallback (lr 1e-3, no scheduler): **byte-identical** `metrics.json`, `mae[0]=0.5209` — fails both the D3 MAE gate (2× the 0.1402 cnn aligned-MAE reference = 0.2804) and, independently, the flat-canvas collapse check.
- val loss frozen at 8.12105e7 from epoch 1; flat amplitude canvas ≈7.97 (std≈1e-6), zero phase.
- The corrected (non-twin) cnn/poisson rung on the **original** aligned data passes cleanly (POISSON-LADDER-001 Rung 1R, mae[0]=0.1662), isolating the defect to the data convention — not the poisson objective and not the cnn/poisson pairing.

### Status
Active (negative control; resolved understanding). Cross-refs: NORMALIZATION-001, DATA-001, RECT-MAE-UNITS-001 (bridge-scale mechanism), POISSON-LADDER-001 (the passing original-data rung).

## POISSON-SCALE-001 - Grid-Lines Dict Path Never Attaches Photon-Count Physics Scale (Opt-In Fix, Not Outcome-Preserving)

> **Scope superseded 2026-07-09:** `count_scale_mode=auto` remains a legacy/amplitude experiment only. It is not the CI calibration mechanism and cannot establish absolute object units. CI uses calibrated count datasets, a physical probe, and physical-mean NLL normalization; see CI-ABSOLUTE-SCALE-CONTRACT-001.
**Category:** PyTorch, Poisson Objective, Units Contract, Bridge Scale, Ablation Methodology
**Impact:** Interpretability of Poisson NLL magnitudes for any grid-lines dict-container run; any future composite objective (hybrid poisson+MAE, regularizers, weight_decay>0) or optimizer/precision change; the `--count-scale-mode` default
**Location:** `scripts/studies/grid_lines_torch_runner.py:1293-1299` (dict-container build), `ptycho_torch/workflows/components.py::derive_dict_physics_scale`, `ptycho_torch/model.py:2290-2291` (both-sides loss-time multiply), `tests/torch/test_dict_container_physics_scale.py`
**Date:** 2026-07-06

### Issue
The grid-lines runner builds a plain dict container that bypasses `_attach_physics_scale`, so `physics_scaling_constant=1.0` in every ladder run (REVISION R2; see POISSON-LADDER-001, RECT-MAE-UNITS-001): the Poisson NLL trained on normalized O(0.01–35) units, not genuine 1e9-photon counts. Consequence: the assumed likelihood is uniformly ~S≈330× under-confident vs the data's actual photon statistics; logged NLL magnitudes are unit-meaningless; any composite objective would mis-weight the data term by ~S²≈1.1e5.

### Fix (opt-in; commits `4cf6d074`, `3305c352`, `8a985df7`)
New runner flag `--count-scale-mode {auto,off}` + `components.derive_dict_physics_scale()`. `auto` attaches `physics_scaling_constant = S` per split, S = `derive_intensity_scale_from_amplitudes(container['observed_images'], nphotons)` (nphotons from NPZ metadata, default 1e9 with warning): S_train=328.696, S_test=329.199 on the aligned data. S derives from `observed_images` (the loss-side raw diffraction, unconditioned in every `--input-conditioning-mode`), not X — X can carry appended conditioning channels that would corrupt S (review finding, fixed in `3305c352`). `off` (the DEFAULT, after the A/B below) leaves the key absent — byte-identical to prior behavior. rms scale is untouched (forward is bit-identical; the lift enters only the both-sides loss-time multiply, `model.py:2290-2291`). Provenance (`count_scale_mode`, `physics_scale_train/test`, `nphotons`+source) is recorded in the run's config.json. Tests: `tests/torch/test_dict_container_physics_scale.py` (5 CPU tests: closed-form S via dataloader wiring; `off` leaves the key absent/collates to 1.0; S sourced from `observed_images` not conditioned X; both-sides-lift identity vs hand-computed Poisson NLL; default = `off`).

### A/B Result — Practical Outcome-Invariance FALSIFIED
Theory: the both-sides lift satisfies `NLL_c(θ)=S²·NLL(θ)+const` exactly, so gradients scale by exactly S² and Adam (verified conditions: single-term objective, fp32, no clip, wd=0) should preserve outcomes. Measured (cnn+poisson 5ep seed 3, exact Phase-1 recipe, current HEAD): `auto` → amp MAE **0.21154**, `off` → **0.16622** (matches the frozen Phase-1 reference 0.1662 to 1.6e-5; final live loss 12667.08 vs recorded 12667). A repeat `auto` run is byte-identical to the first across every per-epoch value ⇒ deterministic, not GPU noise; the `off`-match rules out code drift. Lift verified active in the failing run: provenance S_train=328.6956, live final loss 6.7876e7 (~5358× `off`). The hybrid_resnet rung was not run (red-gate stop).

### Mechanism (Hypothesis — hedged, not confirmed)
Adam's ε=1e-8 is non-negligible only against the unlifted run's smallest per-tensor grad RMS (~1.2e-5), so the lift changes per-step updates by O(0.1%) on those tensors (plus fp32 rounding differences); ~2810 steps of trajectory divergence in the H1-underdetermined (flat) objective select a different basin. Outcome sensitivity to numerically-trivial perturbations is itself further evidence of the POISSON-LADDER-001 flatness finding.

### Practical Rules
1. Default `off` — no existing workflow changes behavior.
2. Lifted and unlifted runs are NOT comparable — never evaluate a lifted run against unlifted references or vice versa.
3. `auto` is the opt-in for units-correct NLL magnitudes (model comparison, UQ, composite-objective calibration) and requires freshly baselined gates.
4. Loss-magnitude expectations: logged loss scales by the operating-point-dependent ratio (~1.2–1.4e4 near convergence, NOT S²≈1.1e5 — S² scales the gradient/θ-dependent part only; Task 7 measured this live).

## TF-LOADER-COORDS-001 - TF Loader Collapses coords_nominal/coords_true to the Same Key, Ignoring True-Position Jitter
**Category:** TensorFlow, Data Contract, Coordinates, Loader
**Impact:** Any dataset that supplies true/jittered scan positions distinct from nominal grid positions (jitter studies, true-position ablations) is silently reduced to the nominal-position case when loaded via `ptycho.loader.load` on the TF training path; `coords_true` never diverges from `coords_nominal` regardless of dataset content.
**Location:** `ptycho/loader.py:466-467`, `ptycho/raw_data.py:124-125,554,589-590`, `docs/specs/spec-ptycho-interfaces.md:78`
**Date:** 2026-07-07

### Issue
`ptycho/loader.py` builds both `coords_nominal` and `coords_true` from the identical dict key:
```python
coords_nominal = dset[key_coords_relative]
coords_true = dset[key_coords_relative]
```
(`key_coords_relative = 'coords_start_relative'`, `ptycho/raw_data.py:125`). Because both variables read the same key, `coords_nominal` and `coords_true` are always numerically identical after this assignment, regardless of whether the underlying NPZ carries separate jittered/true positions. `docs/specs/spec-ptycho-interfaces.md:78` states the requirement only for the *no-jitter* case ("SHALL be equal when jitter/true positions are not provided"), implying divergence otherwise; the implementation does not honor that implication for any input.

### Root Cause
`RawData._generate_dataset_from_groups` (`ptycho/raw_data.py:554,589-590`) separately computes `coords_offsets`/`coords_relative` (from `xcoords`/`ycoords`, i.e. potentially-jittered/true positions) and `coords_start_offsets`/`coords_start_relative` (from `xcoords_start`/`ycoords_start`, i.e. nominal grid positions) into the grouped dict. `ptycho/loader.py`'s `load()` function reads only `key_coords_relative` (`'coords_start_relative'`) for both the `coords_nominal` and `coords_true` container fields — the `coords_relative`/`coords_offsets` pair (which would carry the true/jittered positions when they differ from `coords_start_*`) is never read.

### Impact
True-position jitter datasets are silently ignored by the TF loader path: any downstream code or metric assuming `PtychoDataContainer.coords_true` reflects actual (possibly jittered) scan positions is operating on the nominal grid positions instead, with no error or warning.

### Resolution Options (deferred to user decision)
1. Fix `ptycho/loader.py` to read `coords_relative`/`coords_offsets` for `coords_true` (behavior change; requires new tests to characterize the true-position pipeline).
2. Narrow `docs/specs/spec-ptycho-interfaces.md`'s requirement to state the loader always uses nominal positions for both fields (spec-only fix, accepts current behavior as intended).
